# GetYourOffer — инструкция для команды и AI-агентов

Этот файл — единый источник правды о проекте. Прочитай его целиком, прежде
чем писать код. Поддерживай его актуальным: новые договорённости и решения
фиксируем здесь.

## О продукте

Веб-сервис, который автоматизирует рутину отклика на вакансии для соискателя.
Пользователь вставляет вакансию и своё резюме → получает пакет отклика:
подогнанное под вакансию резюме, анализ соответствия, черновик
сопроводительного письма, запись в трекер откликов.

**Позиционирование (влияет на фичи):** это НЕ «проверка совпадения резюме с
вакансией по ключевым словам» (так делает Jobscan). Продукт закрывает **весь
повторяющийся процесс отклика**, потому что соискатель проходит его десятки
раз за поиск. Каждая фича должна работать на «сделать один отклик за минуту
вместо часа», а не на «показать процент совпадения».

## Аудитория и язык

- Широкий русскоязычный соискатель (не только IT).
- **UI и весь текст интерфейса — на русском.** Бренд английский
  (GetYourOffer), но тексты, кнопки, сообщения об ошибках — по-русски.
- LLM генерирует резюме/письма **на языке вакансии** (русская вакансия →
  русский вывод).
- Код, идентификаторы, комментарии в коде и коммиты — на английском.

## Core-флоу v1 (MUST, без этого продукта нет)

1. Загрузка резюме (PDF) + вставка текста вакансии в поле.
2. Один структурированный вызов LLM, возвращающий JSON (контракт ниже).
3. Рендер результата на чистом дашборде: подогнанное резюме, анализ
   (совпадения + честные пробелы), черновик сопроводительного письма.
4. Сохранение отклика в персональный трекер (куда, когда, статус:
   черновик / отправлено / ответ / отказ / оффер).
5. Аккаунт пользователя (логин) + история откликов и версий резюме.

## LLM: DeepSeek API

- OpenAI-совместимый API, base URL `https://api.deepseek.com`.
- Ключ — только из env `DEEPSEEK_API_KEY` (см. `.env.example`). В код и логи
  ключ не попадает.
- Промпт требует **строго JSON без преамбулы и markdown-обёрток**.

### Контракт вывода LLM (строгий JSON)

```json
{
  "match_score": 0,
  "matches": [
    { "requirement": "", "evidence": "" }
  ],
  "gaps": [
    { "requirement": "", "suggestion": "" }
  ],
  "keywords_to_add": [""],
  "tailored_resume": "",
  "cover_letter": ""
}
```

- `match_score` — 0–100; `matches` — сильные совпадения; `gaps` — чего не
  хватает; `keywords_to_add` — ключевые слова вакансии, которых нет в резюме;
  `tailored_resume` — текст/markdown переточенного резюме; `cover_letter` —
  черновик сопроводительного.
- Парсить ответ безопасно: try/catch, срезать возможные ```json-обёртки.
- Битый/нечитаемый PDF и пустой ответ модели обрабатывать без падения UI.

## Что НЕ делаем в v1

- Парсинг/скрапинг сайтов вакансий и job-бордов — вакансия вставляется текстом.
- Подача заявок за пользователя, интеграции с реальными ATS.
- Тренажёр собеседований (в роадмап, не в v1).
- Браузерное расширение.
- Реальный биллинг. Freemium — **простой счётчик** (3 бесплатные подгонки,
  дальше заглушка «оформить доступ»), без оплаты.

## Приоритет работ

Сначала — **сквозной тонкий срез end-to-end**: загрузил PDF → вставил
вакансию → увидел подогнанное резюме + письмо. Пусть без аккаунтов и трекера,
но от начала до конца. Только потом аккаунты, трекер, история,
freemium-счётчик, полировка дизайна. Не начинать второстепенное, пока
основной путь не работает целиком.

## Данные и приватность

- Резюме = персональные данные. **Не логировать содержимое** резюме, вакансий
  и писем — в логах только метод/путь/статус (см. middleware в
  `backend/internal/transport/http/router.go`).
- Не хранить лишнего, не отправлять никуда, кроме LLM-провайдера.
- Для демо — только своё резюме или обезличенные примеры.

## Структура репозитория

```
getyouroffer/
├── AGENTS.md             # этот файл
├── Makefile              # все команды запуска
├── docker-compose.yml    # Postgres 16 (healthcheck, volume)
├── .env.example          # шаблон конфига; рабочий .env не коммитится
├── backend/              # Go 1.26, отдельный модуль
│   ├── cmd/server/       # entrypoint и composition root: config, pgxpool,
│   │                     #   сборка зависимостей, graceful shutdown
│   ├── internal/         # раскладка ПО СЛОЯМ (layer-first):
│   │   ├── config/       #   чтение env
│   │   ├── models/       #   сущности (User, Profile, Result) + sentinel-ошибки
│   │   ├── service/      #   бизнес-логика + интерфейсы-порты (consumer-side)
│   │   ├── repository/   #   адаптер БД (pgx): UserRepository
│   │   ├── client/       #   внешние клиенты: DeepSeek (LLM), PDFExtractor
│   │   └── transport/http/ # роутер, middleware, хендлеры, WriteJSON/WriteError
│   └── migrations/       # нумерованные SQL: NNNN_name.sql
└── frontend/             # Vite + React + TS + Tailwind v4 + react-router
    └── src/
        ├── lib/          # api.ts (клиент к бэкенду через Vite-proxy),
        │                 #   auth.tsx (JWT в localStorage, AuthProvider,
        │                 #   RequireAuth), quota.ts (freemium-счётчик)
        ├── components/   # Logo, AppHeader, StatusBadge, ScoreRing
        ├── pages/        # Landing, AuthPage, NewResponse, VacancyResult,
        │                 #   Tracker
        ├── App.tsx       # роутер
        └── index.css     # дизайн-токены (@theme): палитра «Маджента»,
                          #   шрифты Oswald/Onest/Ubuntu Mono/Caveat
                          #   (self-hosted через @fontsource)
```

- Бэкенд организован **по слоям** (layer-first). Решение команды от
  2026-07-10, заменяет прежнюю feature-first/hexagonal раскладку
  (`tailor/`, `user/` с `domain/usecase/infra/delivery`):
  - `models/` — сущности и sentinel-ошибки; зависит только от stdlib и uuid.
  - `service/` — бизнес-логика. Здесь же объявлены интерфейсы-порты, которые
    сервис потребляет (`Repository`, `ResumeExtractor`, `ResumeTailor`) —
    **на стороне потребителя**.
  - `repository/` — адаптеры БД (pgx); переводят ошибки драйвера в
    `models`-sentinels, чтобы драйвер не протекал в другие слои.
  - `client/` — исходящие клиенты внешних сервисов (DeepSeek, PDF).
  - `transport/http/` — роутер, middleware и HTTP-хендлеры; там же общие
    хелперы ответа `WriteJSON`/`WriteError` (единый конверт `{"error": msg}`).
    Хендлер зависит от сервиса через свой маленький интерфейс (consumer-side).
  **Правило зависимостей:** `models` импортируют все; `service` импортирует
  только `cmd/server`; конкретные адаптеры интерфейс не импортируют (в Go
  соответствие структурное) — циклов нет. Сборка зависимостей (composition
  root) — в `cmd/server`, роутер только маршрутизирует.
- Без junk-drawer пакетов `util`/`helpers`.
- Миграции: нумерованные идемпотентные SQL (`CREATE TABLE IF NOT EXISTS`),
  применяются по порядку имён через `make db-migrate`. Без инструментов
  миграций — на текущем этапе хватает psql.

## Как запустить

Требования: Go 1.26+, Docker Desktop (запущенный), Node LTS
(`brew install node`).

```bash
make setup       # .env из шаблона + зависимости (go mod download, npm install)
# вписать DEEPSEEK_API_KEY в .env
make db-up       # Postgres :5432, ждёт healthcheck
make db-migrate  # применить backend/migrations/*.sql
make back        # терминал 1: Go API на :8090
make front       # терминал 2: Vite dev на :5173
```

- Порты: бэкенд `:8090`, фронтенд `:5173`, Postgres `:5432` (меняется через
  `POSTGRES_PORT` в `.env` — тогда поменяй и `DATABASE_URL`).
- `GET /api/health` → `{"status":"ok","db":"ok|unavailable|not_configured"}`.
  Бэкенд живёт и без БД (warn в логе) — dev-цикл не зависит от Docker.
- Vite проксирует `/api` на `:8090` (см. `frontend/vite.config.ts`) — фронт
  ходит в API относительными путями, CORS в dev не мешает.

## Конвенции команды

- **Коммиты на английском**: короткий императивный subject (~50–72 символа),
  при необходимости — краткие буллеты в body. **Никаких водяных знаков и
  AI-атрибуции** («Generated with…», «Co-Authored-By: Claude…» — нельзя).
- Пушим прямо в `main`, без PR и CI. Перед пушем: `make fmt vet test` и
  убедись, что фронт собирается.
- `.env` не коммитить; новые переменные окружения сразу добавлять в
  `.env.example` и README.
- README держать актуальным по мере готовности фич (раздел
  «Функциональность»).
- Go: ошибки оборачиваем `fmt.Errorf("...: %w", err)`, не глотаем; при
  маппинге на sentinel сохраняем причину двойным `%w`
  (`fmt.Errorf("...: %w: %w", sentinel, err)`); `context.Context` первым
  параметром у I/O-функций; интерфейсы маленькие (1–3 метода) и объявляются
  **на стороне потребителя**: `service` объявляет порты к `repository`/`client`,
  `transport` — свой порт к `service`.

## Стек, версии и грабли

- **Go 1.26** + `go-chi/chi/v5` (именно `/v5`, не v1!) + `go-chi/cors` +
  `jackc/pgx/v5` (pgxpool).
- **Tailwind v4**: подключается ТОЛЬКО плагином `@tailwindcss/vite` +
  `@import "tailwindcss"` в `index.css`. Никаких `tailwind.config.js`,
  `postcss.config.js`, `npx tailwindcss init`, `@tailwind base/components/
  utilities` — это v3, почти все туториалы в интернете про него. Не
  «чинить» настройку по таким туториалам.
- **docker-compose**: ключ `version:` не пишем (устарел в compose v2);
  в healthcheck `$$` вместо `$` (иначе compose интерполирует сам).
- `.env` парсится Makefile'ом (`-include .env` + `export`): значения без
  пробелов, `#` и `$`.
- Postgres: `postgres:16-alpine`.
